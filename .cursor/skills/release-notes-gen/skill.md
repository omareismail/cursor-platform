# Skill: release-notes-gen

**Invocation:** `/release-notes-gen [version]`

---

## Overview

**Memory references:** `memory-bank/deploymentNotes.md`

`release-notes-gen` generates customer-facing release notes in plain language with zero
implementation detail — separate from the developer-focused `changelog-gen` output.
The audience is end users, product stakeholders, and customer success teams. It rewrites
each changelog item as a user-benefit statement (what the user can now do, not what code
changed), groups items by user-facing area rather than by code layer, and flags any item
requiring product or legal sign-off before publication.

---

## Steps

**Step 1 — Read the internal changelog entry.**

Read the `## [version]` section from `CHANGELOG.md` (generated by `changelog-gen`).

**Step 2 — Rewrite each item in user-benefit language.**

Rules:
- Start with what the user can now do, not what was implemented
- Never mention class names, method names, commit hashes, or technical concepts
- Never mention "fixed a bug in X" — say "we resolved an issue where [user-visible symptom]"
- Use "you can now", "we've improved", "we've resolved" framing

| Internal changelog | User-facing release note |
|-------------------|--------------------------|
| `feat(api): add GET /brokers/{id}/clients/export endpoint` | **Export client data**: You can now download your full client list as a CSV file directly from the Broker Portal. |
| `fix(infrastructure): correct date filtering in client query` | We resolved an issue where the date filter on the client list sometimes showed results outside the selected date range. |
| `perf(application): add Redis caching to product catalog` | Product catalog pages now load significantly faster. |

**Step 3 — Group by user-facing area.**

Group by the feature area the user works in, not by code layer:
- Broker Portal
- Reports & Exports
- Account Management
- Payments & Billing
- [etc. — derived from feature names in the changelog]

**Step 4 — Flag items for product/legal review.**

```
⚠ REQUIRES REVIEW BEFORE PUBLISHING:
  - "Export client data" — involves PII data handling; data team review recommended
  - Any payment flow changes — compliance team review
  - Any price/plan changes — legal review required
```

**Step 5 — Write the release notes file.**

```markdown
# [Product Name] — Release [version] ([Month YYYY])

## What's New

### Broker Portal

**Export client data to CSV**
You can now download your complete client list as a CSV file. Use the Export button
at the top of the Clients page to download all clients, or apply filters first to
export only the clients you need. The export includes all client details, policy
counts, and last contact dates.

**[Other feature]**
[User-benefit description]

## Improvements

- The client list now loads faster, especially for brokers with large client portfolios.
- [Other improvement]

## Fixes

- We resolved an issue where the date filter on the client list sometimes included
  results outside the selected date range.
- [Other fix]

---
*For technical details, see [CHANGELOG.md](CHANGELOG.md).*
```

---

## Example Invocation

**Command:** `/release-notes-gen 1.4.0`

Agent reads `CHANGELOG.md` v1.4.0 entry (3 Added, 2 Changed, 2 Fixed items). Rewrites
all 7 items in user-benefit language. Groups 4 into "Broker Portal", 2 into "Reports &
Exports", 1 into "Improvements". Flags export feature for data team review.

Output: `docs/release-notes/1.4.0.md`

---

## Output

- New: `docs/release-notes/[version].md`
- Terminal: list of items flagged for product/legal review before publishing
